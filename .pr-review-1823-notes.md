## Comment 1 — Already posted (signing order + @available)

Hey there!

I've been working on WebAuthn/FIDO2 support as well and bumped into many of the same problems. A few things / $.02:

### Signing order defeats the entitlement split

greptile flagged this too — the current order signs CLI/helper with `EMBEDDED_ENTITLEMENTS` first, then signs the app `--force --deep` with `APP_ENTITLEMENTS`, which re-signs the nested binaries with the full entitlement set.

https://github.com/manaflow-ai/cmux/blob/f90f17cb40dd93ae1783c1cda8c137a8a850d11c/scripts/build-sign-upload.sh#L98-L103

So long as signing the app bundle first is done things seem happier (`--deep` signs all nested content with full entitlements, then re-sign embedded with narrow). Same fix needed in `nightly.yml` and `release.yml`. Tested in CI:

https://github.com/Jesssullivan/cmux/blob/aba5d140/scripts/build-sign-upload.sh#L95-L107

### `ASAuthorizationWebBrowserPublicKeyCredentialManager` requires macOS 15.0+

The coordinator class instantiates this without availability checks:

https://github.com/manaflow-ai/cmux/blob/f90f17cb40dd93ae1783c1cda8c137a8a850d11c/Sources/Panels/BrowserPanel.swift#L1759-L1763

And the lazy var that creates it:

https://github.com/manaflow-ai/cmux/blob/f90f17cb40dd93ae1783c1cda8c137a8a850d11c/Sources/Panels/BrowserPanel.swift#L1975

This API is macOS 15.0+ only. cmux targets 14.0 — this will crash on Sonoma. Needs `@available(macOS 15.0, *)` on the class and availability checks at all call sites.

Similar SDK-boundary issue with `ASAuthorizationError.Code` cases:

https://github.com/Jesssullivan/cmux/commit/64438e85

I've been testing the signing-order fix and SDK compat guards over here:

https://github.com/Jesssullivan/cmux/actions/runs/23331854667

...and testing the unsigned build against a few FIDO2 implementers (gh, glab, etc). Glad to throw these bits into this PR branch if that'd be desirable 👀

## Comment 3 — DRAFT: CAPTCHA fix (not yet posted)

### Post to: PR #1823 and/or issue #1429

One more thing — I think this PR (and the existing codebase) has a CAPTCHA interaction issue that's likely the root cause of #1429.

The `passkeyAuthorizationBootstrapScriptSource` added here is injected with `forMainFrameOnly: false`:

https://github.com/manaflow-ai/cmux/blob/f90f17cb40dd93ae1783c1cda8c137a8a850d11c/Sources/Panels/BrowserPanel.swift#L2943-L2948

The existing `telemetryHookBootstrapScriptSource` and `addressBarFocusTrackingBootstrapScript` also use `forMainFrameOnly: false`:

https://github.com/manaflow-ai/cmux/blob/f90f17cb40dd93ae1783c1cda8c137a8a850d11c/Sources/Panels/BrowserPanel.swift#L2909-L2923

This means all three scripts run inside cross-origin CAPTCHA iframes (`challenges.cloudflare.com`, `google.com/recaptcha`, `hcaptcha.com`). CAPTCHA providers fingerprint the JS environment inside their iframe, and they detect:

- Overridden `console.log`/`warn`/`error`/`info`/`debug` methods (the telemetry hook replaces all five):

https://github.com/manaflow-ai/cmux/blob/f90f17cb40dd93ae1783c1cda8c137a8a850d11c/Sources/Panels/BrowserPanel.swift#L1747-L1752

```javascript
// CAPTCHA detection: Function.prototype.toString.call(console.log) no longer
// returns "function log() { [native code] }" — it returns the wrapper source
const orig = (window.console && window.console[m]) ? window.console[m].bind(window.console) : null;
window.console[m] = function(...args) {
  __pushConsole(m, args);
  if (orig) return orig(...args);
};
```

- Non-standard `window.__cmux*` globals visible inside the CAPTCHA iframe:

```
window.__cmuxHooksInstalled
window.__cmuxConsoleLog          (512-entry ring buffer)
window.__cmuxErrorLog            (512-entry ring buffer)
window.__cmuxAddressBarFocusState
window.__cmuxAddressBarFocusTrackerInstalled
window.__cmuxPasskeyHooksInstalled  (added by this PR)
```

The fix is straightforward — change `forMainFrameOnly` from `false` to `true` on all three bootstrap scripts. They only need to run in the top-level page context:

- Telemetry: sub-frame console logs aren't needed
- Address bar focus: only applies to the main document
- Passkey authorization: WebAuthn is initiated by the top-level page

I tested this in our fork and it fixes CAPTCHA flows (Cloudflare Turnstile on GitLab specifically):

https://github.com/Jesssullivan/cmux/commit/49a3242c

And for the existing telemetry/focus scripts on `main` (independent of the passkey work):

https://github.com/Jesssullivan/cmux/actions/runs/23352002917

I have both fixes ready as commits on a branch based on this PR if you want them:

https://github.com/Jesssullivan/cmux/compare/upstream/issue-124-passkeys-webauthn...Jesssullivan:cmux:dev/fido2-cherrypicks

This is probably also worth a standalone fix on `main` for #1429 since the telemetry and focus tracking scripts predate the passkey work.

- Complementary to #1823's pre-authorization approach, not a replacement
- Frame as: "entitlement path for signed releases + JS bridge for everything else"

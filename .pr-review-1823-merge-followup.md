## Draft follow-up comment

---

Thanks for the follow-ups in `40fb3b52` — the signing order fix, popup dedup, and origin hardening all look solid.

One thing I still see missing: `BrowserPasskeyAuthorizationManager` and `BrowserPasskeyAuthorizationCoordinator` both use `ASAuthorizationWebBrowserPublicKeyCredentialManager` which is macOS 15.0+ only, but there are no `@available` guards. cmux targets 14.0 — this will crash on Sonoma when the lazy coordinator is first accessed. The new singleton makes it easier to gate:

```swift
@available(macOS 15.0, *)
private final class BrowserPasskeyAuthorizationManager { ... }
```

Plus `guard #available(macOS 15.0, *)` in `configurePasskeyAuthorizationBridge`.

Also — on the broader signing question — we tested the full WebAuthn flow end-to-end on webauthn.io and confirmed that `ASAuthorizationController` (both this PR's `WebBrowserPublicKeyCredentialManager` and our fork's `SecurityKeyPublicKeyCredentialProvider`) requires the app to have a signed application identifier (team ID). Ad-hoc signed builds get:

```
NotAllowedError: The calling process does not have an application identifier.
```

This means contributors testing WebAuthn in fork PRs will hit this unless they have Apple Developer signing set up. Not a bug in this PR — it's an Apple platform constraint — but worth noting for the PR description so reviewers/testers know what to expect.

-Jess

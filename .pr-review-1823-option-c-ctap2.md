## Draft comment for PR #1823

---

Following up on the entitlement discussion from #1876 and this PR.

### The AuthenticationServices entitlement wall (summary)

After testing the full chain, the picture is:

1. `ASAuthorizationController` requires a signed application identifier (team ID). Ad-hoc/unsigned builds get `NotAllowedError: no application identifier`.
2. Even with a team ID, public key credential requests hit Associated Domains enforcement — the app must declare `webcredentials:<domain>` for every relying party. For a general-purpose browser, this is impossible.
3. The only AuthenticationServices escape hatch is `com.apple.developer.web-browser.public-key-credential` (or the broader `web-browser` variant), which requires Apple Developer portal approval and a provisioning profile. The broader variant caused #1739 on macOS 26.

So this PR's approach — restricted entitlement + provisioning profile — is the correct and only path for AuthenticationServices-based WebAuthn. No argument there.

### Complementary path: direct CTAP2 over USB HID

There's a class of users and contributors this doesn't cover:

- Fork maintainers who can't get Apple entitlement approval
- Enterprise environments with managed signing that don't include the restricted entitlement
- Contributors testing WebAuthn changes in PRs (ad-hoc signed dev builds)
- Future cross-platform portability if that's ever on the table

For hardware security keys (YubiKey, SoloKeys, etc.), there's an alternative: talk CTAP2 directly over USB HID via IOKit on macOS (and hidraw on Linux). This bypasses AuthenticationServices entirely — no team ID, no entitlements, no provisioning profile. The tradeoff is it only covers roaming authenticators (USB/NFC keys), not platform authenticators (Touch ID / iCloud Keychain passkeys).

I've built a Zig library at [jesssullivan/zig-ctap2](https://github.com/Jesssullivan/zig-ctap2) (v0.3.0) that handles the full stack: USB HID transport (IOKit/hidraw), CTAPHID framing, CBOR codec, CTAP2 commands (makeCredential, getAssertion, getInfo), PIN protocol v2 (ECDH P-256, AES-256-CBC, HMAC-SHA-256), and structured response parsing — all exported as a C ABI callable from Swift. Zig because cmux already uses it for cmuxd and GhosttyKit.

Tested end-to-end on webauthn.io with YubiKey 5C NFC (registration + authentication, with and without PIN).

The integration model would be:

- **Signed releases:** AuthenticationServices path (this PR) handles everything — platform authenticators + roaming authenticators
- **Unsigned/fork builds:** CTAP2 path handles hardware security keys only, as a fallback when AuthenticationServices is unavailable
- **Detection:** check at runtime whether `ASAuthorizationController` works (team ID present + entitlement available), fall back to direct CTAP2 if not

This isn't a replacement for #1823 — it's a fallback for the builds where AuthenticationServices can't work. The two paths are complementary.

Happy to discuss scope and integration points if there's interest.

-Jess

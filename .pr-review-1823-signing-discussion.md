# PR #1823 — Signing & Application Identifier Discussion

## Draft comment for PR #1823 thread

---

Following up on the earlier review — we've been testing WebAuthn end-to-end and hit something worth discussing.

### `ASAuthorizationController` requires a signed application identifier

Both this PR's approach (`ASAuthorizationWebBrowserPublicKeyCredentialManager`) and our fork's JS bridge approach (`ASAuthorizationSecurityKeyPublicKeyCredentialProvider` + `ASAuthorizationPlatformPublicKeyCredentialProvider`) fail identically on ad-hoc signed builds:

```
NotAllowedError: The operation couldn't be completed.
The calling process does not have an application identifier.
Make sure it is properly configured.
```

Verified on webauthn.io via Safari Web Inspector — the JS bridge installs, intercepts `navigator.credentials.create()`, posts to native, `ASAuthorizationController.performRequests()` fires, and AuthenticationServices immediately rejects it because the app has no team ID in its code signature.

This is separate from the restricted `com.apple.developer.web-browser.public-key-credential` entitlement — it's a base requirement of `ASAuthorizationController` itself. Any app using AuthenticationServices for public key credentials needs to be signed with a real Apple Developer identity (team ID).

### What this means for contributors and fork testers

- **Upstream signed builds** (nightly, release): will work once this PR merges, assuming the signing identity has a team ID ✓
- **Fork builds without Apple Developer signing**: will not work — `ASAuthorizationController` rejects the request before any entitlement check
- **Local dev builds via `reload.sh`**: depend on the developer's local signing identity

This is an Apple platform constraint, not something cmux can work around at the AuthenticationServices level. But it does raise a practical concern: **contributors testing WebAuthn in fork PRs can't verify their changes work** unless they have Apple Developer signing set up.

### The restricted entitlement question

This PR adds `com.apple.developer.web-browser.public-key-credential`, which:

1. Requires Apple approval via the Developer portal
2. Caused launch failures on macOS 26 (#1739) when the broader `web-browser` variant was used
3. Is needed for WKWebView's native WebAuthn (platform authenticators / passkeys)

But `ASAuthorizationSecurityKeyPublicKeyCredentialProvider` (hardware keys like YubiKey) works through `ASAuthorizationController` without that entitlement — it just needs a team ID. So there's a possible layering:

- **Team ID signing only** → `ASAuthorizationController` works → hardware security keys (YubiKey) + passkeys via JS bridge
- **Team ID + restricted entitlement** → WKWebView native WebAuthn also works → pre-authorization bridge (this PR's approach) becomes effective

The first layer has a lower barrier and covers the enterprise FIDO2/YubiKey use case (#1278) without needing Apple's approval for the restricted entitlement.

### Direct USB FIDO2 (entitlement-free)

For fork culture and contributor accessibility, there's also the option of direct CTAP2 communication with security keys over USB HID (via IOKit), bypassing AuthenticationServices entirely. This would work in completely unsigned builds — no team ID, no entitlements. It's more complex to implement but would make WebAuthn testable by any contributor.

Happy to discuss any of these directions — we've been testing various approaches in our fork and have good visibility into what works where.

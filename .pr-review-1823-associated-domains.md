# PR #1823 — Associated Domains Discovery

## Finding

After getting past the application-identifier issue (by adding `com.apple.application-identifier` to entitlements with the correct team ID), `ASAuthorizationController` hits the next wall:

```
Application with identifier QP994XQKNH.com.cmuxterm.app.lab
is not associated with domain localhost
```

This applies to ALL domains — not just localhost. `ASAuthorizationSecurityKeyPublicKeyCredentialProvider` and `ASAuthorizationPlatformPublicKeyCredentialProvider` both require the app to have an associated domain entry (`webcredentials:domain`) for every relying party it authenticates with.

For a general-purpose browser, this is impossible — you can't pre-declare every website.

## What this means

The `ASAuthorizationController` API path (used by both our JS bridge and PR #1823's authorization bridge) **requires** either:

1. `com.apple.developer.web-browser` entitlement — gives browser-level WebAuthn access (crashes macOS 26 without provisioning profile, #1739)
2. `com.apple.developer.web-browser.public-key-credential` entitlement — narrower variant, same provisioning requirement
3. Per-domain associated domains entries — impossible for a general-purpose browser

Safari and Chrome bypass this because they have entitlement #1. PR #1823's approach is correct in requiring the restricted entitlement — there's no middle ground with AuthenticationServices.

## The entitlement hierarchy (complete picture)

```
Ad-hoc signing → no team ID → "no application identifier" error
Developer ID signing → team ID present → "not associated with domain" error
Developer ID + application-identifier entitlement → same domain error
Developer ID + web-browser.public-key-credential → crashes macOS 26 without provisioning profile
Developer ID + web-browser.public-key-credential + provisioning profile → works (PR #1823's target)
Direct CTAP2/USB HID → bypasses all of the above → works in any build
```

## Draft comment for PR #1823 thread

Following up with another finding from testing — after adding `com.apple.application-identifier` to the entitlements (which fixes the "no application identifier" error), the next error is:

```
Application with identifier QP994XQKNH.com.cmuxterm.app.lab
is not associated with domain <rpId>
```

This confirms that `ASAuthorizationController` for public key credentials enforces Associated Domains — the app must declare `webcredentials:<domain>` for every RP. For a general-purpose browser, this means the restricted `web-browser.public-key-credential` entitlement is indeed the only AuthenticationServices path. There's no "just use a team ID" workaround.

This validates your approach in this PR. The restricted entitlement + provisioning profile is the correct path for AuthenticationServices-based WebAuthn.

For contributors testing in forks without the entitlement, the only option is direct CTAP2/USB HID communication (bypassing AuthenticationServices entirely). We're exploring this in our fork as a complementary path.
